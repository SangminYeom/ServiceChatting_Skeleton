# 의사랑 상담 채팅 시스템 설계서

## 개요

의사랑을 사용하는 고객(병원)과 본사 상담사 간의 실시간 온라인 채팅 상담 기능.
기존 "바로바로" 챗봇 웹앱에 상담사 연결 기능을 추가하고, Chatwoot 오픈소스를 상담 엔진으로 활용한다.

## 배경

- **바로바로**: 크로미움 기반 React 웹앱으로 제공되는 챗봇 서비스
- **문제**: 챗봇으로 해결되지 않는 문의 시 전화/원격 외 실시간 상담 채널이 없음
- **목표**: 챗봇에서 상담사로의 자연스러운 전환과 실시간 채팅 상담 기능 제공

## 기술 스택

| 구성 요소 | 기술 |
|-----------|------|
| 고객 앱 (바로바로) | React (크로미움 웹앱) |
| 상담 엔진 | Chatwoot (Docker 자체 호스팅) |
| 기존 백엔드 | .NET (C#) |
| 실시간 통신 | Chatwoot WebSocket (내장) |
| 데이터베이스 | PostgreSQL + Redis (Chatwoot 내장) |
| 파일 저장소 | Chatwoot 내장 (로컬 디스크 또는 S3 호환 스토리지) |

## 시스템 아키텍처

```
고객 (바로바로 React 앱)
  ├── 바로바로 챗봇 UI (기존)
  └── Chatwoot SDK (@chatwoot/sdk) ──→ Chatwoot 서버 (Docker)
                                            ├── WebSocket (실시간 메시지)
                                            ├── REST API
                                            ├── 자동 배정 엔진
                                            ├── 대기열 관리
                                            ├── 파일 저장
                                            ├── CSAT (만족도 평가)
                                            ├── PostgreSQL
                                            └── Redis

기존 .NET 서버
  ├── Chatwoot REST API 호출 (Contact/Conversation 생성)
  └── Chatwoot Webhook 수신 (이벤트 처리)

상담사 (웹 브라우저)
  └── Chatwoot 대시보드 (내장 어드민 패널)
```

### 핵심 구성 요소

1. **고객 앱 (바로바로)**: 기존 챗봇 UI에 "상담사 연결" 버튼 추가. Chatwoot SDK로 상담 채팅 UI 제공.
2. **Chatwoot 서버**: Docker Compose로 자체 호스팅. 상담의 전체 라이프사이클 관리.
3. **기존 .NET 서버**: Chatwoot API를 호출하여 고객/대화 생성, Webhook으로 이벤트 수신.
4. **상담사 대시보드**: Chatwoot 내장 어드민 패널 사용 (별도 개발 불필요).

## 기능 요구사항

### Chatwoot 내장 기능 (개발 불필요)

- 실시간 텍스트 메시지 교환
- 파일/이미지 첨부
- 채팅 이력 저장 및 조회
- 상담사 자동/수동 배정 (라운드 로빈, 가용 상담사 우선)
- 대기열 관리
- 상담 종료 후 만족도 평가 (CSAT)
- 상담사 대시보드 (다중 채팅 동시 처리, 빠른 답변 템플릿, 라벨/분류)
- 오프라인 모드 (업무 시간 외 메시지 남기기)
- Business Hours 설정

### 직접 개발 필요

1. **챗봇→상담사 전환 연동**: "상담사 연결" 버튼 → 사전 정보 수집 → Chatwoot 대화 생성
2. **챗봇 대화 맥락 전달**: 챗봇 대화 내역을 Chatwoot Conversation Note로 자동 전달
3. **고객 인증 연동**: .NET 서버에서 HMAC 토큰 생성 → Chatwoot Identity Validation
4. **Webhook 처리**: Chatwoot 이벤트를 .NET 서버에서 수신하여 비즈니스 로직 실행
5. **Chatwoot 위젯 스타일 커스터마이징**: 바로바로 UI에 맞는 스타일 적용

## 고객 측 UX 흐름

### 온라인 모드 (업무 시간 내)

1. **챗봇 대화 중** — 문제 해결이 안 될 때 "상담사 연결" 버튼 클릭
2. **사전 정보 수집** — 문의 유형 선택 (설치/오류/기능문의/기타)
3. **대기열 진입** — 대기 순서와 예상 대기 시간 표시, 대기 취소 가능
   - 백그라운드에서 .NET 서버가 챗봇 대화 내역 + 고객 정보 + 문의 유형을 Chatwoot에 전달
4. **실시간 채팅** — 상담사가 이미 맥락을 파악한 상태로 대화 시작, 텍스트 + 파일 첨부
5. **만족도 평가** — 상담 종료 후 별점 + 선택적 의견 제출
6. **챗봇 복귀** — 바로바로 챗봇 화면으로 돌아감

### 오프라인 모드 (업무 시간 외)

- "상담사 연결" 클릭 시 안내: "현재 상담 가능 시간이 아닙니다 (평일 09:00~18:00). 메시지를 남겨주시면 영업시간에 확인 후 답변드리겠습니다."
- 고객이 남긴 메시지는 Chatwoot에 대화로 생성, 영업시간에 상담사가 확인 후 응대

## 상담사 측 워크플로우

1. **새 대화 알림 수신** — 자동 배정 또는 대기열에서 수동 수락
2. **챗봇 대화 요약 확인** — 대화창 상단의 노트에서 챗봇 대화 내역, 문의 유형, 고객 환경 정보 파악
3. **실시간 채팅** — 텍스트 + 파일 첨부 + 빠른 답변 템플릿(Canned Response) 활용
4. **해결 완료** — "해결 완료" 버튼 클릭 → 고객에게 만족도 평가 자동 발송
5. **라벨/분류** — 대화에 라벨을 붙여 통계/리포트에 반영

### 상담사 대시보드 구성 (Chatwoot 내장)

- **좌측 사이드바**: 대화, 연락처, 리포트, 설정 메뉴
- **대화 목록**: 내 대화 / 대기 중 / 전체 필터, 문의 유형 라벨 표시
- **채팅 영역**: 상단에 챗봇 대화 요약 노트, 이후 실시간 대화
- **고객 정보 패널**: 병원명, 연락처, 의사랑 버전, OS, 이전 상담 이력

## 연동 설계

### 1) 챗봇 → 상담사 전환

```
"상담사 연결" 클릭
  → 문의 유형 선택 UI 표시
  → .NET 서버에 상담 요청 API 호출 (문의 유형 + 챗봇 대화 내역 포함)
  → .NET 서버 → Chatwoot API: POST /api/v1/contacts (고객 정보 생성/조회)
  → .NET 서버 → Chatwoot API: POST /api/v1/conversations (대화 생성)
  → .NET 서버 → Chatwoot API: POST /api/v1/conversations/{id}/messages (챗봇 요약 노트 전달)
  → 바로바로 앱에서 Chatwoot 위젯 활성화 (conversation_id 연결)
  → 상담사 대시보드에 새 대화 알림 도착
```

### 2) 고객 인증 연동

```
바로바로 앱 로그인 시 고객 정보 확보
  → .NET 서버에 Chatwoot HMAC 토큰 요청
  → .NET 서버에서 Chatwoot Identity Validation용 HMAC 생성 (identifier + secret key)
  → 프론트엔드에 HMAC 토큰 반환
  → Chatwoot SDK 초기화 시 identifier + identifier_hash 전달
  → Chatwoot가 고객을 식별하고 이전 대화 이력 연결
```

### 3) Webhook 연동

```
Chatwoot에서 이벤트 발생
  → .NET 서버의 Webhook 엔드포인트로 POST
  → 이벤트 유형별 처리:
     - conversation_created: 상담 시작 로그
     - conversation_resolved: 상담 종료 처리, 통계 업데이트
     - message_created: 필요 시 메시지 로깅
     - csat_created: 만족도 데이터 수집
```

## 인프라 및 배포

### Docker Compose 구성

| 컨테이너 | 역할 |
|----------|------|
| chatwoot-web | Rails 웹 서버 (API + 대시보드) |
| chatwoot-worker | Sidekiq 백그라운드 작업 |
| postgres | 채팅 데이터 저장소 |
| redis | 캐시 + 큐 + WebSocket pub/sub |

### 서버 요구사항

- CPU: 2코어 이상
- RAM: 4GB 이상
- 디스크: 20GB 이상 (첨부파일 저장 포함)
- Docker + Docker Compose 설치

### 네트워크

```
바로바로 앱 (고객)
  → chat.ubcare.co.kr (Nginx 리버스 프록시)
    → Chatwoot 웹 서버 (:3000)
      → PostgreSQL (:5432)
      → Redis (:6379)

기존 .NET 서버
  ↔ Chatwoot REST API (chat.ubcare.co.kr/api/v1/...)
  ← Chatwoot Webhook (POST)
```

### 초기 설정

1. Inbox 생성 — 웹사이트 채널 타입, 바로바로용 위젯 키 발급
2. 상담사 계정 생성 — 본사 상담사별 계정 및 팀 설정
3. 자동 배정 규칙 — 라운드 로빈 또는 가용 상담사 우선 배정
4. Business Hours 설정 — 평일 09:00~18:00
5. CSAT 활성화 — 상담 종료 시 만족도 조사 자동 발송
6. Webhook 등록 — .NET 서버 엔드포인트 URL 등록

## 에러 처리 및 엣지 케이스

### 연결 장애

| 상황 | 처리 |
|------|------|
| 고객 네트워크 끊김 | Chatwoot SDK 자동 재연결 시도, 대화 유지 |
| 상담사 네트워크 끊김 | 대화 유지, 재접속 시 미읽은 메시지 표시 |
| Chatwoot 서버 다운 | "상담사 연결" 버튼 클릭 시 "일시적 장애" 안내 → 챗봇으로 복귀 |

### 상담 중 예외

| 상황 | 처리 |
|------|------|
| 고객이 대기 중 이탈 | 일정 시간 후 대화 자동 닫힘 (Chatwoot auto-resolve) |
| 고객이 채팅 중 이탈 | 상담사 메시지 전송 → 응답 없으면 대기 후 자동 종료 |
| 상담사 장시간 미응답 | 다른 상담사에게 재배정 (Chatwoot 자동 재배정 규칙) |
| 모든 상담사 오프라인 | 오프라인 모드 전환 → 메시지 남기기 |

### 파일 첨부 제한

| 항목 | 설정값 |
|------|--------|
| 최대 파일 크기 | 40MB (Chatwoot 기본값, 조정 가능) |
| 허용 파일 형식 | 이미지(png, jpg, gif), 문서(pdf, docx), 스크린샷 |
| 업로드 실패 시 | "파일 전송에 실패했습니다. 다시 시도해주세요." 안내 |

### 동시성

| 항목 | 설정값 |
|------|--------|
| 상담사 1인당 최대 동시 채팅 | 설정 가능 (권장: 5~8건) |
| 최대 동시 채팅 초과 시 | 해당 상담사에게 신규 배정 중단, 다른 상담사에게 배정 |

## 규모

- 상담사: 5~20명
- 동시 채팅: 수십 건
- Chatwoot 자체 호스팅은 이 규모에서 충분한 성능 제공
